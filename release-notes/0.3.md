# core 0.3 — PATCH (rebase): `a2a_notifications` rebased onto core 0.2.0

**Version:** 0.3.0 · **Bump:** PATCH (rebase, no wire change) · The
`a2a_notifications` library (originally introduced in core 0.1.0 on a
separate branch) is now rebased onto the core 0.2.0 lineage, producing a
single unified commit on `feat/a2a-notifications` that downstream consumers
(ours-notifications daemon, ours-control-plane) can build against.

No wire shape was changed: all `$field` names in `notify_token_*`,
`webpush_binding_t`, `notify_address_t`, `registration_t`,
`my_registration_t`, and `notification_t` are byte-identical to the core
0.1.0 form. The library surface (`register`, `update_bindings`,
`rotate_token`, `unregister`, `mark_read`, `confirm_registration`,
`post_notification`, `send_notification`, `export_notify_address`,
`set_vapid_public_key`, `export_notify_state`, `import_notify_state`, all
`init` hooks) is unchanged.

The conflict resolution merged the core 0.2.0 additions (`qa_strip_peer_ads`,
`qa_send_restore_request`, `qa_send_fake_restore_response` — contact-survival
tests) with the notification test helpers (`qa_notify_state`, `qa_post_tampered`,
`qa_send_fake_confirm`) so the full T1–T11e + N1–N8 suite now runs against
core 0.3.0 — all green.
