# core 3.0 — Ephemeral-key slim invite

**Version 3.0 · MAJ (breaking) · The fat, identity-bearing invite is replaced by
a slim invite that carries only an ephemeral encryption pubkey; both parties'
address documents (and, for roles, the delegation chain) are exchanged over a
two-message encrypted hop after redemption instead of riding in the invite.**

## What shipped

Invites no longer disclose identity. A core-3.0 invite (`invite_eph_t`,
`a2a_protocol.mm`) carries only `{ invite_id, inviter_container_id, inviter_name,
ephemeral_encryption_pubkey, scheme_id }` — **no public-key list, no
self-signatures, no delegation cert, no root profile.** The inviter's (and, for a
delegated role, the responder's) full address document + chain travel **inside a
box** on a two-message redeem hop, only to a party that has proven liveness by
round-tripping the invite.

The redeem flow is **1 out-of-band invite + 2 on-wire messages** (down from the
old ≈4-message ECDH handshake + accept):

```
  (OOB)  inviter ──► responder : invite_eph_t  { invite_id, inviter_cid, name, eph_pub, scheme }
  leg 1  responder ──► inviter : submit_invite_response   (BARE send; box → inviter eph_pub)
  leg 3  inviter ──► responder : complete_invite          (BARE send; box → responder eph_pub)
```

Both legs are **bare boxed sends** (`_crypto_encrypt_message` to the
counterparty's ephemeral public key — the `authorize_packet` idiom,
`encrypted_channel.mm:118-124`); `encrypted_channel` is **untouched** and resumes
for all post-contact `send_message` traffic once both sides are registered.

The invite is structurally minimal — 5 public fields, no key list / cert / root
profile — so its **size floor is the post-quantum ephemeral public key** (measured
~921 bytes raw, dominated by that key). The dramatic reduction is **vs role
invites**, which previously embedded the root's entire key list + profile + the
delegation chain; a flat invite shrinks too, but the win is largest for roles.

> **Why leg 3 is a bare box, not `encrypted_channel`.** At leg 3 the responder has
> not yet registered the inviter, so an `encrypted_channel::send_encrypted_tx`
> reply is undecryptable receiver-side — the responder cannot resolve the inviter's
> source key (`key_storage.mm:291` "Unknown source key for message decryption",
> reproduced in loopback). Boxing to the responder's **real** long-term key is also
> impossible: the stdlib never exposes a raw long-term encryption private key. So
> leg 3 boxes to the responder's **ephemeral** public key (carried in leg 1); the
> responder keeps its ephemeral private key until leg 3 to open it. This corrects
> the plan's original "leg 3 via encrypted_channel / box to the real key" wording.

After leg 3 both sides hold the other's verified AD + contact + (optional) root
linkage — symmetric to the pre-3.0 end state.

**Disclosure order flips:** the responder now discloses its identity first (boxed
to the inviter's ephemeral key). This is inherent — moving the identity bulk off
the OOB channel forces recipient-first disclosure — and is neutral under the
OOB-trust model (see *Security notes*).

## New/changed transactions (`a2a_messaging.mm`)

- **`generate_invite($name)`** — unchanged signature; now mints the slim
  `invite_eph_t` via the reworked `mint_eph_invite` (was `mint_invite`). Generates
  a per-invite ephemeral encryption keypair: the public half ships in the invite,
  the private half is stored in the new hidden `pending_invite_keys`.
- **`add_contact($invite, $name?)` → LEG 1 (responder).** Parses the slim invite,
  generates a one-shot responder ephemeral keypair, boxes its identity bundle
  (`{ad, cert?, root_profile?, cp_binding?, invite_id}`) to the invite's ephemeral
  pubkey, and **bare-sends** `submit_invite_response`. Records `pending_redemptions`
  (chosen name + expected inviter cid) and keeps the responder ephemeral private
  key in the new hidden `pending_redemption_keys`. The contact is **pending** here;
  it finalizes on leg 3. Returns `$resp_eph_pub` (already-public wire material) so a
  caller can correlate the outstanding redemption.
- **`submit_invite_response` (NEW, library-routed) → LEG 2 (inviter).** Inbound,
  bare box. Gates in order — pending-invite lookup → box decrypt → cid-bind
  (`AD.container_id == envelope.from`) → `process_address_document(ad, TRUE)` (PoP
  self-sig) → optional delegation-chain verification — **then** registers the
  responder and **single-use consumes both** `pending_invites` and
  `pending_invite_keys`. Replies leg 3 as a bare box to the responder's ephemeral
  pubkey. Deliberately does **not** `check_encrypted_or_abort` (the box is the
  protection).
- **`complete_invite` (NEW, library-routed) → LEG 3 (responder).** Inbound, bare
  box. Gates — pending-redemption lookup → inviter-cid pin (`from == expected
  inviter`) → box decrypt with the kept responder ephemeral key → cid-bind →
  `process_address_document(ad, TRUE)` → optional chain — **then** registers the
  inviter and clears **both** `pending_redemptions` and `pending_redemption_keys`.
  Also bare (no `check_encrypted_or_abort`).

State (`a2a_messaging.mm`): `pending_invites` changed from `(global_id ->> str)` to
`(global_id ->> pending_invite_t)` (`{assigned, eph_pub, scheme}`); new hidden,
**non-exported** `pending_invite_keys` and `pending_redemption_keys` (ephemeral
private keys); new `pending_redemptions` (`{inviter_cid, inviter_name,
custom_name}`).

## Breaking changes

- **Invite format + redemption flow changed; old fat invites are NOT redeemable
  on 3.0, and 3.0 invites are NOT redeemable by pre-3.0 clients.** Invites are
  short-lived artifacts; re-issue them after upgrading. (Clean break, plan §7
  option A.)
- **Removed:** `a2a_protocol::invite_t`, `a2a_protocol::invite_role_t`,
  `a2a_protocol::rebuild_peer_address_document`, and the old offline-redeem
  `add_contact` body. (`accept_contact`/`handle_accept_contact` are retained as a
  legacy inbound handler; they are no longer reachable from the 3.0 redeem path.)
- **Outstanding invites do not survive a daemon restart** (see *Security notes* —
  this is by design, fail-closed).

## Integration TODO for consumers (later phases — NOT in this core-only change)

- `add_contact` is now **asynchronous**: it returns a *pending* status; the contact
  finalizes when leg 3 arrives. Consumers that treated `add_contact`'s return as
  "contact added" must instead surface completion on the leg-3 `notify_agent`
  (`$contact_added`) event. The inviter's confirmation arrives via the leg-2
  `$contact_accepted` event.
- No new `::actor::` shim is required: `submit_invite_response` / `complete_invite`
  are library-routed (`::a2a_messaging::…`), like `ingest_connect_descriptor`.
- The slim invite blob is much smaller; consumer compression/transport is
  unaffected (the blob is still an opaque bin).
- Re-issue any outstanding invites after the upgrade.

## Security notes

- **Identity pin is unchanged.** Both legs enforce `received_AD.container_id ==
  envelope.from` **and** `process_address_document(ad, TRUE)` (proof-of-possession
  self-signature) before storing or trusting any AD — identical to the pre-3.0
  `handle_accept_contact` gate. Pin strength still tracks the planned `container_id`
  key-derivation; neither improved nor regressed.
- **No registration before all gates pass (INV-5).** Neither leg writes
  `contacts`/`peer_ads`/`contact_roots` until lookup → decrypt → cid-bind → PoP →
  chain all pass. A failed gate mutates nothing.
- **Single-use.** The first valid leg 2 consumes both inviter stores
  (`pending_invites` + `pending_invite_keys`); the first valid leg 3 consumes both
  responder stores. A replayed or invalid leg consumes nothing.
- **Reduced disclosure / smaller blast radius.** An intercepted invite leaks only
  `{eph_pub, cid, name}` — no keys, no root profile, no org chain. A leaked invite
  forces an active, single-use round-trip (revealing the attacker's own cid) rather
  than being passive total identity exposure.
- **Ephemeral secret at rest, and restart-transience BY DESIGN.** The per-invite
  ephemeral private keys live in `pending_invite_keys` / `pending_redemption_keys`,
  which are `hidden` **and never added to `export_core_state` (INV-4)**. The export
  blob is the host's persistence boundary (`export_state` → disk → `import_state`),
  so an outstanding invite/redemption **does not survive a daemon restart** between
  legs — it fails closed (the next leg aborts on the missing key; no contact is
  silently formed). Treat invites as **session/process-transient**; re-issue after a
  restart.
- **`AdaptValue.Serialize` corrupts raw secret keys — LANDMINE.** Empirically (M0
  spike), a raw `secretkey_encrypt` round-tripped through the SDK
  `AdaptValue.Serialize()` / `ParseValue()` path (the host's `export_state` →
  on-disk blob, `index.ts:1296`) comes back **present-but-unusable** (decrypt
  aborts "crypto box failed"). It DOES round-trip faithfully through mufl's own
  `_write`/`_read`. This is a second, independent reason the ephemeral secret stores
  must never be placed in `export_core_state`: doing so would both leak the secret
  AND yield a corrupt, silently-failing key on reload. Do not "fix" restart-survival
  by adding these stores to the export.
- **Eph-key TTL — RECOMMENDED, NOT implemented in this change.** Bounding the
  at-rest-secret window and stale-invite growth wants an expiry sweep over
  `pending_invites`/`pending_invite_keys` (and the responder stores). It needs a
  time source + a sweep hook and is deferred (plan §10). Until then, the stores are
  bounded only by single-use consumption and restart-transience.
- **Leg 1 is an unauthenticated bare inbound.** Anyone holding a leaked invite can
  spam `submit_invite_response`, each forcing one decrypt attempt; it aborts cheaply
  on the lookup / box-open / cid-bind, mutates no state before the gates, and the
  first valid one consumes the invite. Bounded DoS; documented, accepted.
- **MITM exposure unchanged.** Both designs are TOFU rooted in the OOB invite
  channel; an active OOB attacker can substitute the whole invite either way, and
  the defense is the same (verify the cid you pinned, out of band).
