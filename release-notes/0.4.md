# core 0.4 — MINOR: `a2a_notifications` v2 — per-contact notification tokens

**Version:** 0.4.0 · **Bump:** MINOR (new protocol surface, fully additive — no wire-shape change,
no format_version bump) · Delivers the per-contact scoped-token model on top of the v1
notifications library. All v1 senders and recipients continue to work without modification;
v2 behaviour is opt-in and upgrade-path-driven.

---

## What changed

**Per-sender scoped tokens via `$scope`.** The reserved `$scope` field in `notify_token_core_t`
(always `""` in v1) is now used to mint one token per (recipient, sender) pair: a scoped token
is a standard v1 token whose `$scope` is set to `_str(sender_cid)`. Token version stays `1`;
`notify_address_t` stays `v1`; byte-equality validation already covers `$scope`, so no
wire-shape change is required. A scoped token inside an unchanged handout blob works with
un-upgraded senders for free — they treat the blob as opaque.

**New service-side inbounds (all encrypted-channel, registered-recipient gate):**

- `issue_tokens ($senders -> global_id[]+)` — mint or return (idempotent, E8 discipline) a
  scoped token for each requested sender CID; capped at `issue_max_senders = 256` per call
  (the client wrapper pages larger lists). Returns the full per-sender map in the confirm.
- `set_sender_muted ($sender -> global_id, $muted -> bool)` — toggle the receive-mute flag for
  one sender at the service, runtime-only (no token change, no redistribution). `$muted FALSE`
  deletes the flag; absent = enabled. Re-confirms so the client mirror stays coherent.
- `revoke_sender_tokens ($senders -> global_id[])` — delete each sender's scoped token from the
  index and from `notify_sender_tokens`, without re-minting. Posts using those token IDs abort
  at the index lookup (E4). Idempotent: unknown senders tolerated.
- `retire_shared` — delete the shared (scope-`""`) token's index entry, rendering existing
  shared-handout posts dead at step 3. The registration record stays; scoped tokens and
  re-confirms keep working. Idempotent (second call is a no-op).

**`rotate_token` gains optional `$sender -> global_id`:**

- `$sender` present → rotate ONLY that sender's scoped token; all other tokens untouched.
- `$sender` absent/NIL → v1 behaviour for the legacy shared token PLUS rotate ALL scoped slots
  (panic button; every old handout dies instantly, E4 semantics). Rotate-all respects the
  retired state: if the shared token was already unindexed via `retire_shared`, the new shared
  token is also left unindexed — rotate-all does not un-retire a deliberately retired token.

**Confirm extension (additive).** `confirm_registration` now carries two extra fields:

- `$sender_tokens -> (global_id ->> notify_token_t)` — full current per-sender map for the
  recipient. Old clients reading only `$token/$vapid_pub/$bindings` ignore these fields safely.
- `$sender_muted -> (global_id ->> bool)` — full current mute map.

New clients store the per-sender map wholesale into `my_notify_contact_tokens[service]` on each
confirm — re-confirms are idempotent replays.

**`post_notification` validation extended for scoped tokens.** The 7-step validation now
dispatches on `$scope` after the index check. The scoped path adds sender binding (envelope
`$from` must match `$scope` — abort `"Token is not bound to this sender."`) and a mute check
(`notify_sender_muted` present entry — abort `"Notifications from this sender are disabled."`).
See the contract doc v2 section for the full ordered list.

**New state maps (all additive; empty by default; individually guarded on import):**

- Service: `notify_sender_tokens (global_id ->> (global_id ->> notify_token_t))`,
  `notify_sender_muted (global_id ->> (global_id ->> bool))`.
- Client: `my_notify_contact_tokens (global_id ->> (global_id ->> notify_token_t))`.

`export_notify_state`/`import_notify_state` gain all three fields with the existing
optional-read guard pattern — a v1-era export imports unchanged; the new maps default to empty.

**New client-side transactions:**

- `notify_issue_tokens ($service, $contacts -> global_id[])` — pages into batches of 256 automatically.
- `notify_set_sender_muted ($service, $contact -> global_id, $muted -> bool)`.
- `notify_revoke_contact_tokens ($service, $contacts -> global_id[])`.
- `notify_retire_shared ($service)`.
- `notify_rotate_token` gains optional `$contact -> str+` (resolved via `resolve_contact`).

## Back-compat matrix

| Recipient version \ Sender version | v1 sender (un-upgraded) | v2 sender |
|---|---|---|
| **v1 recipient** | unchanged (deployed today) | v2 sender posts the shared handout verbatim — valid, blob is opaque to senders |
| **v2 recipient** | v1 sender holds the old shared handout — valid until `notify_retire_shared` is called; after retirement posts abort at index lookup | full v2 per-contact flow |

The shared token continues to be minted and indexed on every new registration; retirement is an
explicit, grace-windowed step — existing deployments are unaffected until they opt in.

---

## Migration (from deployed core 0.3 / NaaS PoC)

This is a **mufl-compatible (additive)** bump: no existing `$field` is renamed or retyped, and
`format_version` does not change. The migration rides the normal blob-boot lane:

1. Daemon upgrades first (new inbound names; old client code simply never calls them).
2. App upgrade lands; host detects a bound service with empty `my_notify_contact_tokens` and
   runs a backfill: `notify_issue_tokens(all contacts)` → distribute per-contact blobs → ledger
   tracks sent/acked per contact.
3. Contacts receive per-contact handouts as they come online. The shared handout stays valid
   during the overlap period.
4. After all non-degraded contacts have acknowledged (or after a 30-day grace window), the host
   may call `notify_retire_shared` to permanently kill the legacy shared handout.

Rollback is safe: additive fields are ignored by old code on both halves.

---

## Non-goals (restated from v1; no change in v2)

Payload content remains **plaintext to the service** (transport legs are encrypted, but the
service decrypts and stores the notification). There is **no rate limiting or anti-spam** beyond
per-contact token revocation and rotation; the recovery path for abuse is `rotate_token ($sender)`
or, for a leak of unknown origin, the rotate-all panic button.
