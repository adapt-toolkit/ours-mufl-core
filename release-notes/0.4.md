# core 0.4 — `a2a_notifications` v2: per-contact notification tokens (shared token removed)

**Version:** 0.4.0 · **Bump:** breaking rewrite of the notifications library (no deployed
v1-notifications clients exist; consumers are (re)built against 0.4). Per-contact scoped
tokens are the ONLY model — the v1 shared-token feature is deleted, not deprecated.

---

## What changed

**Per-contact scoped tokens only.** Every token names exactly one (recipient, sender) pair:
`$scope == _str(sender_cid)`. Token `$version` stays `1` and `notify_address_t` stays
version `1` (the `$scope` field always existed in the shape). Tokens are minted exclusively
by `issue_tokens`; **registration mints nothing**.

**Shared-token feature removed entirely:**

- `registration_t` and `my_registration_t` no longer carry a `$token` field; the
  `confirm_registration` targ no longer carries `$token`.
- `handle_register` stores the registration record only — no mint, no index entry.
- `retire_shared` / `notify_retire_shared` (a transitional retirement path drafted during
  development) do not exist.
- `post_notification` has a single validation path: index lookup → **sender binding**
  (envelope `$from` must equal the token's `$scope` — abort
  `"Token is not bound to this sender."`) → byte-equality against
  `notify_sender_tokens[recipient][sender]` → mute check (abort
  `"Notifications from this sender are disabled."`) → payload cap.
- `rotate_token` with `$sender` absent rotates ALL scoped slots (panic button);
  with `$sender` present rotates exactly one.
- `unregister` is a full teardown: registration, every scoped token (slot + index entry)
  and the mute map are purged.

**Service-side inbounds (all encrypted-channel, registered-recipient gate):**

- `issue_tokens ($senders -> global_id[]+)` — mint or keep (idempotent, E8, byte-stable) a
  scoped token per requested sender cid; capped at `issue_max_senders = 256` per call
  (the client wrapper pages larger lists). Re-confirms with the full per-sender map.
- `set_sender_muted ($sender -> global_id, $muted -> bool)` — runtime receive-mute toggle
  (present entry = muted, absent = enabled). No token change, no redistribution.
- `revoke_sender_tokens ($senders -> global_id[])` — delete each sender's scoped token from
  the index and `notify_sender_tokens`, without re-minting (the contact-removal path).
- `rotate_token` gains optional `$sender -> global_id` (per-sender vs rotate-all).

**Client-side transactions:**

- `notify_issue_tokens ($service, $contacts -> global_id[])` — pages into batches of 256.
- `notify_set_sender_muted ($service, $contact -> global_id, $muted -> bool)`.
- `notify_revoke_contact_tokens ($service, $contacts -> global_id[])`.
- `notify_rotate_token` gains optional `$contact -> str+` (resolved via `resolve_contact`).
- `export_notify_address ($service, $contact)` — exports the PER-CONTACT handout blob from
  `my_notify_contact_tokens[service][contact]`; aborts if issuance hasn't confirmed yet.

**Confirm shape.** Every service-side mutator replies the same idempotent confirm:
`($vapid_pub, $bindings, $sender_tokens, $sender_muted)`. The client replaces
`my_notify_contact_tokens[service]` wholesale on each confirm (single source of truth;
re-confirms are idempotent replays). The `on_notify_registration` hook now receives
`$sender_tokens` and `$sender_muted` beside `$service_cid`/`$registration`, so a consumer
(the messenger's distribution engine) can diff/mirror per-contact state.

**State maps:**

- Service: `notify_sender_tokens (global_id ->> (global_id ->> notify_token_t))`,
  `notify_sender_muted (global_id ->> (global_id ->> bool))`.
- Client: `my_notify_contact_tokens (global_id ->> (global_id ->> notify_token_t))`.

`export_notify_state`/`import_notify_state` carry all fields with the individually-guarded
optional-read pattern — a pre-0.4 export imports with the new maps empty.

## Security model (why per-contact)

| Property | shared token (removed) | per-contact (0.4) |
|---|---|---|
| Leak blast radius | any leak ⇒ anyone can spam the recipient | one relationship's channel only |
| Attribution | none — all contacts hold identical bytes | `$scope` names the relationship |
| Authorization | possession alone | possession + sender binding (`$from == $scope`) — a stolen handout is useless without the named sender's signing key |
| Recovery granularity | rotate-all + full redistribution | rotate/revoke one contact; rotate-all retained as panic button |
| Per-relationship control | none | receive-mute at the service (unprobeable), plus consumer-side send gating |

---

## Deployment

There is no migration path from the 0.3 shared-token PoC — deployments are recreated:
the service daemon ships first (new inbound names), clients register + `issue_tokens`
against it. Old handout blobs from any pre-0.4 deployment die at validation (their
token_ids are not in the new service's index; unscoped tokens additionally fail sender
binding).

---

## Non-goals (unchanged)

Payload content remains **plaintext to the service** (transport legs are encrypted, but the
service decrypts and stores the notification). There is **no rate limiting or anti-spam**
beyond per-contact rotation/revocation.
