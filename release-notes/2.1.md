## What shipped

Core-side protocol support for **CP-brokered cluster introductions**. A root
binds a control plane (CP) once; that single bind:

- registers the CP as the manager of the root's whole cluster (CP side), and
- lets every child of the root **inherit** the root-signed CP binding so the
  child accepts the CP's introductions locally, with no per-child bind ceremony.

The CP can then introduce child components of *different* roots to each other
(the existing root-agnostic `introduce` / `introduce_to_group`); the introduced
parties do fresh end-to-end DH, so the CP never holds their channel keys.

Application logic required is still just the `core.connect` boolean ‚Äî everything
else is identity-hierarchy + control-plane material the host/daemon manages.

This is **additive**: the legacy direct-bind introduction path (a node that ran
its own 6-digit proxy bind) is unchanged and still accepts introductions.

## New/changed transactions (`a2a_messaging.mm`)

- **`manage_root($root_cid)`** ‚Äî host-fired (CP side). Records that this control
  plane manages a root (`managed_roots` set). The authorization half of
  enrollment: a child enrolls only if its delegation chain resolves to a root
  here. Idempotent.
- **`set_root_cp_binding($binding)`** ‚Äî host-fired. Stores the root-signed CP
  binding (`root_cp_binding`), verified against the node's own root identity
  before storage. Fills the long-standing gap (the binding was restored on
  import but never settable). It then rides the existing invite / state-push
  paths down to the root's roles.
- **`enroll_delegated_node`** ‚Äî external inbound (CP side). A root relays one of
  its children's signed address document + delegation chain; the CP verifies
  **(1)** the authenticated sender *is* the root in the chain, **(2)** that root
  is in `managed_roots`, **(3)** `verify_peer_delegation` chains the child to it,
  and re-checks the child AD self-signature (proof-of-possession). On success it
  stores `peer_ads`/`contacts`/`contact_roots`. Idempotent.
- **`require_cluster_cp_or_abort`** (helper) replaces `require_bound_cp_or_abort`
  in `ingest_connect_descriptor`'s relay gate. Additive: accepts **either** a
  legacy direct bind (`monitoring_proxy == sender`) **or** an inherited cluster
  CP (sender is the CP the node's root designated, re-verified locally against
  the pinned `root_ad` keys ‚Äî including the `cid_cp` pin ‚Äî on every call).
- `managed_roots` threaded through `export_core_state` / `import_core_state`
  (optional; pre-2.1 exports default to empty).
- `version.mm`: `2.0 ‚Üí 2.1`.

`a2a_protocol.mm` is unchanged ‚Äî the existing hierarchy/attestation shapes and
verifiers (`delegation_cert_t`/`verify_peer_delegation`,
`root_cp_binding_t`/`verify_root_cp_binding`, `cp_attestation_t`) were already
sufficient.

## Integration TODO for consumers (later phases, NOT in this change)

- **MCP daemon** (`ours-mcp/packages/core`): at root-bind, relay each
  subagent's AD to the CP via `enroll_delegated_node`; call `manage_root` for the
  bound root; push `root_cp_binding` to roles via the role-spawn state push; run
  the `get_manifest` `core.connect` pre-check before calling `introduce`.
- **Control-plane / messenger**: UX for the cluster-introduction flow.
- **Monitoring inheritance** is deliberately **NOT** changed here: a root bind
  does **not** auto-copy all roles' traffic to the CP. That is a governance
  decision (it would change the monitoring guarantee's semantics) and is parked
  unless explicitly approved.
- **Recursive delegation** (N-level trees) is parked; the enrollment + gate are
  written so a later chain-walk could slot in, but flat 2-level is all that ships.

## Security notes

- This change **promotes the previously non-enforcing governance-attestation
  layer** (`root_cp_binding` / `cp_attestation` / invite-threading) to an
  **enforcing** authorization role gating introductions. The crypto is sound ‚Äî
  root-signed, domain-separated, locally pinned via `root_ad` ‚Äî and the residual
  trust (the host honestly distributes `root_ad` / `root_cp_binding` to its
  roles) is the same honesty class as the existing monitoring self-assertion
  limit. **The promotion decision is the founder's**; implemented on a branch,
  not merged to `main` until signed off.
- Invariants preserved: the node-side capability gate (`self_supports(core.connect)`)
  stays authoritative; `enroll_delegated_node` rejects a child of an unmanaged
  root (and a relayer that is not the root); `require_cluster_cp_or_abort`
  re-verifies the binding against the pinned `root_ad` keys (and `cid_cp`) every
  call; the `process_address_document(peer_ad, TRUE)` self-signature re-check
  stays in `ingest`.
