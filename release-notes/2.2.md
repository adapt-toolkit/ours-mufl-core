# core 2.2 — Cluster root-side producers

**Version 2.2 · MIN (additive) · The root-side PRODUCER half of cluster
CP-introduction: a root can now MINT its CP binding, push it to its roles, and
RELAY its children to the control plane. Completes the loop core 2.1 left to
"later phases".**

## What shipped

core 2.1 shipped the cluster RECEIVER + ACCEPT side (`manage_root`, the
`enroll_delegated_node` handler, the `set_root_cp_binding` store,
`require_cluster_cp_or_abort`, and `$rpb` propagation) but deferred the
root-side PRODUCERS to consumer integration — and no transaction existed to
(a) mint the root-signed `root_cp_binding` or (b) emit the enrollment relay to
the CP. The managed/root side therefore could not actually drive enrollment.
core 2.2 adds those producers, all additive and root-side:

- A root **mints** its §3c CP binding (`sign_root_cp_binding`) — self-signing
  `{context_tag, root_cid, cid_cp}` via the same key path as delegation certs.
- The daemon stores it on the root AND pushes the same blob to each role;
  `set_root_cp_binding` is widened so a role verifies its **inherited** binding
  against its pinned `root_ad` (the accept gate's own keys + verifier), letting a
  role hold the edge with zero ceremony.
- A root **relays** each child's signed AD + delegation chain to the bound CP
  (`relay_enroll_delegated_node`), driving the existing CP-side
  `enroll_delegated_node` handler — one root bind enrolls the whole cluster.

No CP-side or node-accept-side behaviour changed; the legacy direct-bind path is
untouched.

## New/changed transactions (`a2a_messaging.mm`)

- **`sign_root_cp_binding($proxy)`** — NEW, host-fired, ROOT only, STATELESS.
  Resolves `$proxy` to the CP's container id, builds
  `root_cp_binding_core_t{version 1, context_tag "ours:cp-attestation:v1",
  root_cid=self, cid_cp}`, root-signs it via `key_storage::default_sign` over
  `_value_id`, and returns the signed blob (+ `cid_cp`). Mirrors
  `sign_delegation`. Aborts if not a root (`delegation_cert != NIL`).
- **`set_root_cp_binding($binding)`** — CHANGED (additive verify-path widening),
  now DUAL-MODE. ROOT (`delegation_cert == NIL`): verify the binding against the
  node's own identity — unchanged from 2.1. ROLE (`delegation_cert != NIL`):
  verify the INHERITED binding against the pinned `root_ad` — rejecting any edge
  not signed by the role's own root. Fails CLOSED. This is a NEW, tightly-gated
  acceptance, not a narrowing: in 2.1 the role path always aborted (it
  own-identity-verified a root-signed edge, which can never match), so 2.2 newly
  accepts root-signed inherited bindings a role could not store before. The root
  path is byte-unchanged; no pre-existing accept path is loosened.
- **`relay_enroll_delegated_node($proxy, $child_ad, $delegation_cert,
  $root_profile)`** — NEW, host-fired, ROOT only. Resolves `$proxy` to the bound
  CP and sends an encrypted `enroll_delegated_node` relay carrying
  `{$child_ad (AD value), $delegation_cert (blob), $root_profile (blob)}` —
  field-for-field matching `handle_enroll_delegated_node`. Returns the relay
  target + child container id. Aborts if not a root.
- **`version.mm`**: `2.1 → 2.2`.

`a2a_protocol.mm` is unchanged — `root_cp_binding_t` / `verify_root_cp_binding` /
`cp_attestation_context_tag` and the delegation verifiers were already
sufficient.

## Integration TODO for consumers (MCP daemon)

- At root↔CP bind: `sign_root_cp_binding(cp)` then `set_root_cp_binding(blob)` on
  the root; push the same blob to each role via `set_root_cp_binding` (role path,
  `root_ad`-verified).
- For each child: `relay_enroll_delegated_node(cp, child_ad, cert, root_profile)`.
- `manage_root` / `introduce` / `introduce_to_group` and the `get_manifest`
  `core.connect` pre-check stay CP-side (messenger), unchanged.
- The child container id the CP keys on is `child_ad.identity.container_id`; the
  daemon must source it from ONE place and keep it byte-consistent with the cid it
  advertises elsewhere (e.g. `list_agents`).

## Security notes

- These are PRODUCERS, not new enforcement. The root signs its OWN governance
  edge with the same key it uses for delegation certs; the relay is an
  authenticated root→CP send over the existing bound channel; and the CP handler's
  three checks (sender is the child's root, that root is managed, the delegation
  chain verifies + AD proof-of-possession) are unchanged. No trust-model change
  beyond making the 2.1 layer usable from the root side.
- `set_root_cp_binding` EXTENDS the verify semantics of the
  governance-attestation enforcement layer: a role now stores an inherited edge,
  verified against its pinned `root_ad` (`binding.root_cid` == `root_ad`'s
  container id; signature under the domain-separated `context_tag`; `cid_cp`
  integrity via the signed core record). The role path is a NEW acceptance, not a
  narrowing: in 2.1 a role's `set_root_cp_binding` always aborted (own-identity
  verify of a root-signed edge can never match), so 2.2 newly accepts root-signed
  inherited bindings the role could not store before — tightly gated to edges
  signed by the role's own root, matching the accept gate's trust. The root path
  is byte-unchanged. Additive (MIN); no pre-existing accept path is loosened.
- Like core 2.1's promotion of the governance-attestation layer to an ENFORCING
  role, this change rides the **founder promotion gate**: implemented on a branch
  (`feat/cluster-root-producers`), NOT merged to `main` until signed off. The
  producers further activate that enforcing layer (roles can now inherit and
  accept CP introductions end-to-end), so founder visibility applies.
