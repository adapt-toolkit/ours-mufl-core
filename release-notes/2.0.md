# ours mufl core — Configuration / Control-Plane release notes

Covers the generic configuration & control-plane features added to the shared
mufl core on branch `feat/core-config-control-plane` (core **1.10 → 2.0**).
Written for the engineers who will later wire these into the TS wrappers
(MCP daemon, browser messenger) and other ours apps. **No app integration is
included in this branch** — this is the protocol layer only.

> Status: implemented + compiles clean against the ADAPT toolkit. B1/B3 code
> reviewed GREEN by critic-1; B1 verified by ours-tester-1 (22/22 runtime +
> hostile-write + cold-channel). **B2 was radically simplified in core 2.0**
> (SAS / CP-signature / connect-descriptor machinery removed — see the B2 section);
> the simplified design reviewed GREEN by critic. No open questions for the
> dev/test track (the `outgoing_only` direction question is moot — policy was
> removed from the protocol entirely).

| Core | Feature |
|------|---------|

| 1.12 | **B1 hardening** (hidden gate state, ceremony relocated into `a2a_messaging`) + **B3** — control-plane configuration (`set_app_config`). |

bumps; a node compiled at an older core stayed wire-compatible. **Core 2.0 is a
MAJ bump**: it removes wire shapes and changes the `ingest_connect_descriptor`
payload, so consumers of the old `core.connect` surface must update (see
*Breaking changes* below). The monitoring/config surface is unchanged across 2.0.

## Already present before this work (context)

The manifest + capability surface the brief asked for already existed:

- `a2a_capabilities::app_manifest_t` = `(version, app_id, name, description, monitoring_status, capabilities)`, read via the readonly **`get_manifest`** transaction. `app_id` is also auto-stamped on every `a2a_control` message.
- Typed capabilities: `core.configuration`, `core.monitoring`, `core.connect`; verb envelope `(cap, verb, args, req_id)` + `dispatch`.
- `core.configuration` carries an **opaque JSON `$params`** (schema + values the frontend renders) with **secret-field redaction** (`secret_field_t`, `$set/$unset/$needs_reentry`, epoch auto-clear on CP change).
- `a2a_control` transport: `send_control(contact, payload)` + inbound `control_message` (origin + known-sender gated, `on_control_received` hook).
- Connect primitives existed in an earlier form (`connect_descriptor_t` + `verify_connect_descriptor`, `safety_number` SAS); **all removed in core 2.0** in favour of relaying peer-signed address documents directly (see the B2 section).

This work makes those usable as a **generic** control plane: forced monitoring,
a place to store/enforce config, and an introduction flow.

## B1 — Forced monitoring

**Goal:** once a control plane (CP) is bound, the node is protocol-forced to
re-encrypt and forward a copy of every chat message to it, and the app cannot
turn that off.

### Where the enforcement lives (and why)
- Copy **generation** is unconditional core code in `a2a_messaging`’s
  `send_message` and `handle_receive_message` (the single path every app routes
  through) — **not** an app-injected hook (an app could no-op a hook).
- The **gate state** (`monitoring_proxy`, `proxy_pending`) is declared `hidden`
  in `a2a_messaging`. `hidden` blocks external read **and** write, so no app or
  other library can do `a2a_messaging::monitoring_proxy -> NIL` to disable
  monitoring. Because hidden state is mutable only by its declaring library, the
  bind ceremony + disable live **in `a2a_messaging`**.
- Copies are **fire-and-forget through the broker**: one
  `encrypted_channel::send_encrypted_tx` to the proxy, no local queue, no
  liveness handling. If the CP is offline the ADAPT broker holds the message.
- **Uniform rule:** `monitoring_proxy` bound ⇒ emit; else nothing. No
  role-vs-root sink split.
- **No recursion:** the copy rides a distinct tx (`::a2a_monitoring::receive_monitoring_copy`), never `send_message`, so copies aren’t themselves monitored.
- **Re-encrypted:** ciphertext on the wire under the CP channel key.

### Transactions (node side — `a2a_messaging`)
| Transaction | Origin | Args | Effect |
|---|---|---|---|
| `set_proxy_pending` | user (host) | `$code: str`, `$proxy: str (contact ref)` | Store the host-generated 6-digit code for a contact. |
| `verify_proxy_code` | user (host) | `$code: str`, `$sender: str (contact ref)` | Atomic verify (≤3 attempts, 300 s expiry, failures returned as **data** not aborts). On success sets `monitoring_proxy`. |
| `disable_monitoring` | **external** | — | CP-AUTHENTICATED: sender must be the bound proxy. Clears the binding. No user/app-callable clear exists. |
| `get_monitoring_status` | readonly | — | `($monitored, $proxy_pending, $proxy_cid)`. |
| `monitor_copy_actions` | (internal fn) | direction, peer_cid, date, body | Builds the fire-and-forget copy; self-gates on `monitoring_proxy`. |

### Transactions (control-plane side — `a2a_monitoring`)
| Transaction | Origin | Args | Effect |
|---|---|---|---|
| `receive_monitoring_copy` | external | `$copy: monitoring_copy_t` | Known-contact-only; delegates storage to the `on_monitoring_copy_received` init hook. |

`monitoring_copy_t = (version, direction "out"/"in", peer_cid, peer_name, date, body)`.

### Wrapper interaction
1. The CP becomes a contact of the node (normal invite, Ring 3).
2. Host/daemon runs `set_proxy_pending(code, cpContact)` (the daemon generates the 6-digit code, shows it host-side).
3. Human enters the code in the CP UI → CP’s bind reaches the daemon → daemon calls `verify_proxy_code(code, cpContact)`.
4. From then on every chat send/recv emits a copy to the CP. The CP packet handles `receive_monitoring_copy` and stores the feed via `on_monitoring_copy_received` (call `a2a_monitoring::init` with that hook).
5. To stop: the CP sends `disable_monitoring` (only it can).

**Scope (v1):** copies cover **chat send/receive only** — contact establishment
and control traffic are out of the "all traffic" scope.

## B3 — Control-plane configuration (opaque only)

**Goal:** the bound CP configures the app from the rendered JSON schema. The core
**stores and transports** an opaque, app-custom config blob and bakes in **no
policy semantics whatsoever** — per-application policy (including anything like
"outgoing-only") is **application logic in the wrapper**, driven by the app's own
custom config schema, NOT a protocol feature (human ruling, 2026-06-17).

The bound CP (`monitoring_proxy`) is the configuration authority. `set_app_config`
is **external + CP-authenticated** via `require_bound_cp_or_abort` (sender must be
the bound proxy). `app_config` is **hidden** (so an app can’t rewrite its own
stored config).

### Transactions (`a2a_messaging`)
| Transaction | Origin | Args | Effect |
|---|---|---|---|
| `set_app_config` | external (CP) | `$config: str` | Store the **opaque** JSON config blob (the core never parses it). Fires `$config_updated`. |
| `get_app_config` | readonly | — | `($config)`. |

That is the **entire** B3 protocol surface: opaque config storage + arbitrary
per-app JSON schemas. No typed policy, no core-enforced semantics.

### MUFL vs wrapper split
- **Everything app-specific → wrapper.** `app_config` is owned by the TS wrapper.
  On `$config_updated`, the wrapper pulls `get_app_config` and applies *all* of it
  — operational config (e.g. telegram-connector: bot tokens, chat→contact routing)
  AND any policy the app wants (e.g. "outgoing-only"). The core never maps
  JSON↔mufl and enforces nothing app-specific.
- The core's only job here is to **transport + store** the opaque blob (hidden,
  CP-gated, persisted) and notify the wrapper to re-apply.

### Wrapper interaction
1. CP renders `core.configuration`’s `$params` schema, user fills it.
2. CP sends `set_app_config(json)` → node stores it, fires `$config_updated`.
3. Wrapper pulls `get_app_config` and applies it (operational + any app policy).

## B2 — `core.connect` control-plane introduction (core 2.0, simplified)

**Goal:** the CP, bound to two nodes that aren’t in each other’s contacts,
introduces them (e.g. set node B as a proxy for telegram-connector A).

**Design (radically simple).** The CP already holds each managed node’s
peer-signed address document in `peer_ads` (captured when the node bound the CP).
An introduction is therefore just: **send each node the other’s signed AD.** The
receiving node verifies the AD’s own self-signature, that the relay came from its
bound CP, and that its own manifest still advertises `core.connect`, then registers
the contact immediately. There is **no SAS, no confirmation step, and no CP
signature on the introduction** — the bound-CP channel *is* the authorization, and
the node-side capability gate is the authoritative “do I accept introductions”.

### Flow
1. Daemon pre-check (CP side): pull `get_manifest` for A and B; **refuse to
   introduce if EITHER lacks `core.connect`** (prevents a half-open pairing). This
   is daemon logic, not core.
2. CP calls `introduce(A, B)` (host-fired, user origin). The core looks up
   `peer_ads[A]` and `peer_ads[B]` and emits, in one transaction, two encrypted
   relays to `ingest_connect_descriptor`: A receives B’s signed AD (+ B’s CP-supplied
   display name) and B receives A’s.
3. Each node’s `ingest_connect_descriptor` (external, encrypted):
   - `require_bound_cp_or_abort(sender)` — the relay must come from my bound CP;
   - **node-side capability gate** — abort unless `a2a_capabilities::self_supports("core.connect")`
     (reads the live `describe()` manifest). This is the authoritative enforcement.
   - `process_address_document(peer_ad, TRUE)` — re-checks the peer’s **own**
     self-signatures (proof-of-possession; aborts on a forged/inconsistent AD before
     any write — `TRUE` only relaxes external-authorizing-container trust, not the
     self-signature);
   - registers the peer (contact + `peer_ads`) under the CP-supplied name (cid as a
     last resort) and fires `$introduced` (`container_id, name, by_cp`).
   - **Idempotent re-introduction:** if the peer is already a contact, it only
     refreshes the stored AD and keeps the existing name (`$reintroduced` — never a
     downgrade or rename).

`introduce_to_group(joiner, members)` is the cluster-root fan-out: for each member
it emits the same pair of relays as `introduce(joiner, member)`, all in one
transaction — the natural “new subagent joins, give it everyone” primitive.

### Transactions (`a2a_messaging`)
| Transaction | Origin | Args | Effect |
|---|---|---|---|
| `introduce` | user (CP host) | `$peer_a: str`, `$peer_b: str` (contact refs) | Emits the two signed-AD relays to A and B. Aborts if either is not an established contact (no `peer_ads`) or if `a == b`. |
| `introduce_to_group` | user (CP host) | `$joiner: str`, `$members: str[]` | Emits `introduce(joiner, member)` relays for every member, one transaction. |
| `ingest_connect_descriptor` | external (CP) | `$peer_ad: t_address_document`, `$peer_name: str` | Node-side: CP-sender gate + node capability gate + AD self-sig (PoP), then register; `$introduced` / `$reintroduced` notify. |

The node-side inbound is the dedicated tx `::a2a_messaging::ingest_connect_descriptor`
(NOT routed through the `a2a_capabilities` verb dispatcher).

### Security notes
- **`core.connect` in the manifest means “I ACCEPT introductions” — not “my AD is
  never relayed / I am undiscoverable”.** A node’s address document is public
  (invite-equivalent), so the boolean does not hide it. The half-open preventer is
  the **daemon pre-check**, which must refuse if *either* party lacks support (not
  just the recipient). The node-side gate then independently enforces acceptance.
- The CP-supplied `$peer_name` is an **unauthenticated** display label (receiver-
  chosen); the AD self-signature is the only authenticated identity the receiver
  trusts. Never key an enforcement decision on the name.
- Trust model: the node trusts its **bound CP** to introduce honestly (no SAS).
  This is deliberate — the same governance/honesty class as the monitoring bind and
  the README §8 self-assertion limit, documented not oversold.

### Wrapper interaction
The daemon orchestrates: (1) `get_manifest(A)` + `get_manifest(B)`, refuse if either
lacks `core.connect`; (2) call `introduce(A, B)`; (3) the core emits both signed-AD
relays; (4) each node’s `ingest_connect_descriptor` gates and registers. The CP
never holds the A↔B channel keys (fresh DH between A and B on first message).

## Out of scope / deferred

- **Governance-attestation layer** (`cp_attestation_t` 1.8 / `root_cp_binding_t`
  1.9 / invite threading 1.10): kept **off the critical path** per the human’s
  ruling — out-of-scope-pending-founder-review. B1/B2/B3 do not depend on it.
- Frontend per-app-name rendering (B4) — a different repo (messenger).
- **Per-application policy (e.g. "outgoing-only") — explicitly OUT of the
  protocol** (human ruling). The core provides opaque config + arbitrary per-app
  schemas; it bakes in no app-specific semantics. Apps implement policy in the
  wrapper from their own config.
- Generic JSON↔mufl config auto-mapping.
- App integration / daemon wiring in any consuming repo.

> Note: the standalone `signed_intro_t` CP-signed introduction path (briefly
> required and implemented in core 1.15 as `mint_cp_introduction` /
> `verify_cp_introduction`) was **removed in core 2.0** when B2 was simplified to
> bound-CP-channel authorization. The underlying `intro_t` / `signed_intro_t` /
> `book_entry_t` shapes remain in `a2a_protocol` — they are the out-of-repo
> local-book registrar credential, now unused inside this repo (kept; prune later
> if confirmed dead).

## Breaking changes in core 2.0 (B2 simplification)

Consumers of the old `core.connect` surface MUST absorb:

- **`ingest_connect_descriptor` payload changed**: `{$descriptor (connect_descriptor_t blob), $intro (signed_intro_t blob)}` → `{$peer_ad (t_address_document), $peer_name (str)}`.
- **`$introduced` notify shrank**: it no longer carries `$sas` or `$monitoring_status` — just `$container_id, $name, $by_cp`. Any agent/UI reading `$sas` must update.
- **Removed transactions**: `get_connect_descriptor`, `mint_cp_introduction`, `confirm_introduction`, `list_pending_introductions`. The messenger’s SAS / pending-confirm UI and any flow calling these must be reworked (introductions are now established immediately, with no confirm step).
- **New CP-side transactions**: `introduce(A, B)`, `introduce_to_group(joiner, members)`. The daemon drives these after its `get_manifest` pre-check (refuse if either party lacks `core.connect`).
- **`a2a_capabilities::init` is now a hard prerequisite on the ingest path**: the node-side gate calls the live `describe()` hook via `self_supports`, so an uninitialised `describe` **fails closed** (aborts the ingest). “Supports introductions” is `core.connect` present in `describe()`’s live manifest `$capabilities` map — NOT the `init` `$supported` list.

## Integration TODO for consumers (when this lands)

- The monitoring bind ceremony + status moved to **`::a2a_messaging::`**
  (`set_proxy_pending`, `verify_proxy_code`, `disable_monitoring`,
  `get_monitoring_status`) — update any daemon that addressed them at
  `::a2a_monitoring::`.
- A control-plane packet must call `a2a_monitoring::init($on_monitoring_copy_received)`
  to receive the feed.
- `config.mufl` now exports `a2a_monitoring`; consumers’ core checkout must include it.
- **`core.connect`**: the daemon must `get_manifest`-pre-check both parties and call
  `introduce` (see *Breaking changes* above); the node packet must wire
  `a2a_capabilities::init` (describe hook advertising `core.connect`) or it will
  reject all introductions.

## Open questions

None outstanding for the dev/test track.

- **`outgoing_only` direction** — moot: the typed contact policy was removed
  entirely (human ruling 2026-06-17). Per-app policy is application logic in the
  wrapper, driven by the app's own custom config schema; the core stores only the
  opaque blob.
- **B2 authorization model** — RESOLVED (core 2.0): the non-repudiable CP-signed
  introduction (`signed_intro_t`, core 1.15) was **removed** when the human ruled
  for a radically simpler design. Authorization is now the bound-CP channel itself
  plus the node-side `core.connect` capability gate; no SAS, no per-introduction
  signature. (The 1.15 CP-signed path was the prior decision, now superseded.)
